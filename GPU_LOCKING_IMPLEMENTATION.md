# GPU Locking Implementation - Pessimistic Concurrency Control

**Date:** May 5, 2026  
**Problem:** Race conditions when multiple clients submit jobs simultaneously for the same GPU  
**Solution:** Pessimistic database-level locking using SQLAlchemy's `with_for_update()`

---

## Problem Scenario

```
Timeline (Without Locking):

T1 - Job A submitted
  └─ Query: "Find online GPUs"
     └─ Result: GPU-X (status=online, locked_until=null)

T2 - Job B submitted (milliseconds later)
  └─ Query: "Find online GPUs"
     └─ Result: GPU-X (same GPU! both queries ran before lock)

T3 - Job A assignment
  ├─ job_a.gpu_id = GPU-X
  ├─ GPU-X.locked_by_job_id = job_a
  └─ GPU-X.locked_until = now + 30s

T4 - Job B assignment
  ├─ job_b.gpu_id = GPU-X  ← DOUBLE BOOKING! ❌
  ├─ GPU-X.locked_by_job_id = job_b (overwrites job_a)
  └─ GPU-X.locked_until = now + 30s

Result: Both jobs think they have exclusive access to GPU-X
```

---

## Solution: Pessimistic Locking

```
Timeline (With Pessimistic Locking):

T1 - Job A submitted
  └─ Query: "Find online GPUs" + FOR UPDATE
     └─ Database locks GPU-X row at read time
     └─ Job A acquires lock

T2 - Job B submitted (milliseconds later)
  └─ Query: "Find online GPUs" + FOR UPDATE
     └─ Database WAITS for Job A's lock to release
     └─ Blocked until T3 completes

T3 - Job A assignment
  ├─ job_a.gpu_id = GPU-X
  ├─ GPU-X.locked_by_job_id = job_a
  ├─ GPU-X.locked_until = now + 30s
  └─ COMMIT → Job A releases lock

T4 - Job B's query resumes
  └─ Query sees: GPU-X (locked_until = now + 30s, which is > now)
     └─ GPU-X SKIPPED (already locked)
     └─ Finds next available GPU (GPU-Y)

Result: No double-booking, jobs get different GPUs ✅
```

### 2. Locking Functions

**File:** `backend/app/services/matching.py`

#### `find_available_gpu_and_lock()`
```python
async def find_available_gpu_and_lock(
    db: AsyncSession,
    job_id: str,
    min_vram: int = 0,
    lock_duration_seconds: int = 30
) -> GPU | None:
    """
    - Uses .with_for_update() for database-level pessimistic lock
    - Only considers GPUs where:
      - status == online
      - locked_until is NULL or locked_until <= now (expired lock)
    - Automatically acquires lock for this job on return
    - Lock expires in 30 seconds (configurable)
    """
```

**How it works:**
```sql
-- SQL generated by with_for_update()
SELECT * FROM gpus
WHERE status = 'online'
  AND (locked_until IS NULL OR locked_until <= now)
ORDER BY vram_mb ASC
LIMIT 1
FOR UPDATE;  -- ← Locks the row until commit/rollback
```

#### `unlock_gpu()`
```python
async def unlock_gpu(gpu: GPU) -> None:
    """Clear the lock fields"""
    gpu.locked_by_job_id = None
    gpu.locked_until = None
```

#### `cleanup_expired_locks()`
```python
async def cleanup_expired_locks(db: AsyncSession) -> int:
    """
    Periodically runs to clean up expired locks
    - Finds all GPUs with locked_until <= now
    - Clears their lock fields
    - Returns count of cleaned GPUs
    """
```

### 3. Job Submission Flow

**File:** `backend/app/routers/jobs.py`

**For specific GPU selection:**
```python
if gpu_id:
    # User requested a specific GPU
    result = await db.execute(
        select(GPU)
        .where(GPU.id == gpu_id)
        .with_for_update()  # Lock it immediately
    )
    gpu = result.scalar_one_or_none()
    
    if gpu and gpu.status == GPUStatus.online:
        now = datetime.now(timezone.utc)
        if gpu.locked_until is None or gpu.locked_until <= now:
            # Lock is available, acquire it
            gpu.locked_by_job_id = job_id
            gpu.locked_until = now + timedelta(seconds=30)
        else:
            # Already locked, skip
            gpu = None
    
    # If specific GPU locked, fall back to auto-match
    if not gpu:
        gpu = await find_available_gpu_and_lock(db, job_id)
```

**For auto-matching:**
```python
else:
    # No specific GPU, use locking function
    gpu = await find_available_gpu_and_lock(db, job_id, min_vram=0)
```

### 4. Background Cleanup Task

**File:** `backend/app/main.py`

```python
async def _cleanup_gpu_locks_background():
    """Runs every 60 seconds to clean up expired locks"""
    while True:
        try:
            await asyncio.sleep(60)
            async with AsyncSessionLocal() as session:
                cleaned_count = await cleanup_expired_locks(session)
                if cleaned_count > 0:
                    print(f"🧹 Cleaned up {cleaned_count} expired GPU lock(s)")
        except asyncio.CancelledError:
            break
        except Exception as e:
            print(f"⚠️  Error in cleanup: {e}")
```

**Started in lifespan:**
```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    cleanup_task = asyncio.create_task(_cleanup_gpu_locks_background())
    app.state.cleanup_task = cleanup_task
    yield
    app.state.cleanup_task.cancel()
```

## How Pessimistic Locking Works

### Database Lock Levels

| Lock Type | Behavior | Use Case |
|-----------|----------|----------|
| `FOR UPDATE` | Blocks other transactions from reading/writing | Exclusive lock needed |
| `FOR SHARE` | Blocks writes, allows other reads | Shared lock |
| No lock | Allows concurrent reads/writes | Read-only queries |

### Transaction Isolation

```
Thread 1                           Thread 2
├─ SELECT ... FOR UPDATE gpu_id=1
│  ├─ Lock acquired on gpu_id=1
│  └─ (Other threads wait here)
├─ UPDATE gpu SET locked_by_job=A  
│  (locked_until = now + 30s)
└─ COMMIT
   └─ Lock released ──────────────→ SELECT ... FOR UPDATE gpu_id=1
                                     ├─ Lock acquired (was waiting)
                                     ├─ Checks: locked_until = past time
                                     ├─ Skips this GPU (already locked)
                                     └─ Finds GPU-2 instead
```

---

## Lock Duration

| Event | Time | Lock Status |
|-------|------|---|
| Job A assigned to GPU-X | T=0s | `locked_until = now + 30s` |
| Cleanup runs | T=60s | Checks: 30s < 60s? YES → Unlock |
| Job A completes | T=90s | GPU becomes `online` again |

**Default:** 30 seconds (from `find_available_gpu_and_lock(lock_duration_seconds=30)`)

**Why 30 seconds?**
- Long enough for job assignment + worker connection
- Short enough that hung connections don't block GPUs forever
- Backup cleanup runs every 60 seconds

| Issue | Solution |
